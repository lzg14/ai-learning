# Ext05：AI时代的产品重构——从功能到委托

> 当软件载体消失，能力在泛化，科技公司的产品还是软件吗？本文系产品经理对 AI 时代产品形态演变的深度思考，探讨从功能点到任务、从操作到委托、从 SaaS 到 SaaS（Service as a Software）的根本性转变。

---

## 一、用户从操作者变成委托者

长久以来，我们默认产品是给人用的。人点按钮、填表单、走完操作路径，亲自完成每一件事。

现在，越来越多的产品在被 Agent 调用——Agent 读 API、传参数、拿结果、判断要不要继续下一步，它不需要看见界面。

**人的角色跟着变了。** 过去人亲自做事，现在人表达目标、检查关键节点、承担最终责任——从**操作者**变成了**委托者**。

这意味着：产品设计要从「帮助人操作」转向「帮助人委托」。

> 作者生造了一个词：**Agent-readable**——正在变成和 user-friendly 同等重要的产品标准。

---

## 二、Agent-readable：新一代产品设计语言

以往，优秀的产品要让人看得懂、找得到、点得动。业界曾提出"眯眼法则"：优秀的产品，用户在眯着眼睛完全看不清的前提下，也能知道该怎么用。

而如果一个产品是给 Agent 设计的，那么它几乎不需要考虑配色、动态交互——而需要有：

- **清晰的能力描述**
- **输入输出 schema**
- **权限边界**
- **错误返回**
- **上下文说明**

---

## 三、界面：从操作台到监控台

| | 过去（给人用） | 未来（给 Agent 用） |
|---|---|---|
| UI 作用 | 帮人一步步完成操作 | 帮人看见进度、确认节点、回滚操作 |
| API 粒度 | 动作级（`create_order`） | 任务级（`prepare_customer_meeting`） |
| 体验本质 | 路径 | 协议 |

> **软件给人用时体验是路径，给 Agent 用时体验是协议。**

传统 API 一个接口做一个动作，Agent 更需要任务级接口——一个接口完成一整件事。

---

## 四、Skill：方法的产品化

### 提示词 vs Skill

- **提示词**：一段文本，告诉模型怎么回答问题
- **Skill**：一个打包的 `.zip`，包含任务边界、执行步骤、工具说明、输入输出格式、示例、脚本、模板、错误处理和质量标准

一个好的 Skill 是流程、知识、工具、模板和评估标准的混合体——更像一个轻量级的安装包，中心从**代码**换成了**方法**。

### Service as a Software

通过 Skill，方法本身可以产品化了：

- 咨询公司的方法论
- 律师的审查框架
- 编辑的改稿流程
- 运营的复盘模板

这些都可以变成 Skill 直接分发。过去各种工具只能封装高度标准化的流程，现在连需要判断力的方法也可以封装。

> **过去：SaaS（Software as a Service）→ 未来：SaaS（Service as a Software）—— 服务业的软件化，从这里打开了新入口。**

Skill 是一次性软件和长期软件之间的中间形态，每次执行都可以生成不同的过程和结果——**Skill 是可复用的过程生成器**。

---

## 五、Skill 的三层支撑：Memory + Eval + Permission

光有 Skill 不够，还需要三者配合：

### Memory：长期差异化的来源

功能可以复制，界面可以模仿，模型可以替换——但一个系统长期积累的用户上下文、项目历史、组织规则、失败案例，很难瞬间搬走。留住用户的关键，是系统越来越懂你怎么做事。

### Eval：测试系统 + 进化引擎

没有 Eval，Agent 的进步无法被证明。没有 Eval，Skill 的升级没有方向。没有 Eval，企业不敢把核心流程交出去。

### Permission：Agent 产品体验的核心

用户最关心的不只是 Agent 能做什么，还有它会不会越界。能读什么、能改什么、能发什么、能删什么、哪些地方必须问人——这些边界要让用户看得见。

---

## 六、产品进化路径：从 Feature 到 Capability

| 维度 | 过去 | 未来 |
|------|------|------|
| 核心工作 | 加功能（Feature） | 加能力（Capability） |
| 验证方式 | MVP（最小可执行版本） | **MVT**（最小可完成任务） |
| 规划文档 | Feature Roadmap | Capability Roadmap |
| 交互节点 | 按钮 | 确认 |
| 交付单位 | 功能点 | Skill |
| 传统软件的交互节点是按钮，Agent 软件的交互节点是确认。用户不需要每一步都参与，但关键判断必须可审查。|

> Skill 改了就可能影响大量 Agent 任务。因此 Skill 需要版本管理、回滚、灰度、测试和审计——未来会出现 **SkillOps**，和今天的 DevOps、MLOps 一个逻辑。

## 七、OPC → NPC：零人公司

### OPC（一人公司）

一个人通过各种 AI 工具，能够产生匹敌一个团队的生产力——人提出目标，Agent 执行。本质是自动化的增强版。

### NPC（零人公司）

如果 OPC 成立，为什么不做 NPC？

- 库存数据异常 → 系统自己发起补货任务
- 客户流失风险升高 → 系统自己发起挽回动作
- 项目延期概率上升 → 系统自己调整资源

没有人下达指令，系统根据数据、环境和目标函数（增长目标、成本目标、风险目标、效率目标）自己判断该做什么。

> **Agent 不再围绕用户请求运行，开始围绕目标状态运行。**

---

## 八、权限问题：从执行权限到意图权限

过去管的是 Agent 能不能发邮件、能不能改文件。未来还要管它能不能**主动发现目标、主动创建任务、主动影响资源分配**——Agent 治理从**执行权限**升级到**意图权限**。

最终形态可能是：一个会自我生成任务的 Agent 执行网络。用户看不见页面，甚至看不见任务提出的过程，看到的只是一个组织持续运行后的状态变化。
