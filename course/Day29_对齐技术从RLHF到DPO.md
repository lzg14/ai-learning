# Day 29：对齐技术从 RLHF 到 DPO

> 怎么让大模型"说人话、做好人"？本课深入对齐技术的原理、RLHF 的完整流程、DPO 为什么能替代 RLHF、各种对齐方法的对比。

## Part 1：什么是对齐？为什么需要对齐？

### 对齐的核心问题

**"听话又能干"类比：**

| 没有对齐的模型 | 对齐好的模型 |
| --- | --- |
| 你问它 A，它答 B，答非所问 | 准确理解你的意图，答你想要的 |
| 问它敏感问题，它直接回答 | 拒绝回答有害问题，价值观正确 |
| 说话像机器人，很难懂 | 说人话，流畅自然，懂上下文 |
| 你让它写代码，它写散文 | 准确执行指令，输出符合要求 |

> 💡 对齐（Alignment）的定义：让模型的行为符合人类的意图和价值观。
> 预训练只是让模型"会说话"，对齐才是让模型"说人话、听人话、做好人"。

---

### 对齐的三个核心目标

| 目标 | 说明 | 例子 |
| --- | --- | --- |
| **1. 有帮助 (Helpful)** | 能帮用户解决问题，听懂指令 | 你让写 Python，不写 Java |
| **2. 诚实 (Honest)** | 不胡说八道，不编造，不知道就说不知道 | 不懂的问题不瞎编答案 |
| **3. 无害 (Harmless)** | 不输出有害、偏见、歧视、危险内容 | 不教人造炸弹，不歧视任何人 |

> 这三个目标有时候是冲突的。太诚实了可能会有害，太无害了可能没帮助。对齐就是找三者的最佳平衡点。

---

### 没有对齐的模型是什么样的？

| 问题 | 表现 |
| --- | --- |
| **指令跟随差** | 你让它总结，它给你续写；你让它写代码，它给你讲概念 |
| **输出风格奇怪** | 像读书、像论文，就是不像人说话 |
| **敏感问题不拒绝** | 问它怎么自杀，它真的给你列方法 |
| **充满偏见** | 说程序员都是男的，护士都是女的 |
| **幻觉极其严重** | 张口就来，编得比真的还真 |

## Part 2：RLHF 原理与完整流程

### RLHF = Reinforcement Learning from Human Feedback

**"老师教学生"类比：**
1. **SFT 有监督微调**：老师先给学生看范文，告诉学生怎么答题
2. **训练奖励模型**：老师教给大家"什么是好，什么是坏"
3. **PPO 强化学习**：学生答题，老师打分，学生根据分数改进

---

### RLHF 完整三步曲

#### 第一步：SFT（有监督微调）

**目标：让模型学会跟着指令走**

```plaintext
输入：大量的 <prompt, response> 成对数据
   ↓
用这些数据继续训练基模型
   ↓
输出：SFT 模型，能听懂基本指令了
```

**数据要求：**
- 几万到几十万条高质量的人工标注数据
- 覆盖各种场景：问答、写作、代码、创意、推理等
- 数据质量 >> 数据数量，1 万条好数据 > 100 万条垃圾数据

**效果：**
- 解决 80% 的基础问题：能听懂指令，输出格式基本对
- 剩下 20% 的"好不好"的问题，SFT 解决不了

---

#### 第二步：训练奖励模型（Reward Model）

**目标：学会给回答打分，替代人工打分**

```plaintext
输入：同一个问题，两个不同的回答
   ↓
人工标注：哪个回答更好？A 比 B 好，还是 B 比 A 好？
   ↓
训练一个奖励模型：输入 prompt + response，输出 0-1 的分数
   ↓
输出：奖励模型 RM，能自动判断回答的好坏
```

**核心要点：**
- 标注数据量：几十万对偏好数据
- 奖励模型一般和基模型同大小或者小一点
- 这一步是 RLHF 最关键的一步，奖励模型不准，后面全白费

---

#### 第三步：PPO 强化学习

**目标：用奖励模型当老师，让模型越答越好**

```plaintext
SFT 模型初始状态
    ↓
1. 模型生成一个回答
2. 奖励模型给这个回答打分
3. 如果分数高，就奖励这个生成路径
   如果分数低，就惩罚这个生成路径
4. 循环几百万次
    ↓
最终模型：生成的回答越来越符合人类偏好
```

**PPO 的三个损失项：**
1. **奖励损失**：要让奖励模型打的分越高越好
2. **KL 散度损失**：不能和原来的 SFT 模型差太远，不然会忘东西
3. **语言模型损失**：保持模型本身的语言能力不下降

---

### RLHF 的问题和痛点

| 痛点 | 说明 |
| --- | --- |
| **1. 太复杂** | 三个阶段，每个阶段都有一堆坑，工程难度极大 |
| **2. 太贵** | 几十万条人工标注，百万级 PPO 步，几百万美元起 |
| **3. 不稳定** | PPO 训练非常容易崩，奖励黑客、模式崩溃、忘能力 |
| **4. 奖励黑客** | 模型学会了讨好打分器，输出看起来好实际没用的内容 |
| **5. 可复现性差** | 不同团队做出来效果天差地别，看运气 |

## Part 3：DPO——直接偏好优化

### DPO 为什么是革命？

**"一步到位"类比：**
- RLHF：先学打分，再用打分强化学习 → 两步，麻烦，不稳定
- DPO：直接把人类偏好塞到模型里 → 一步，简单，稳定

> DPO 论文出来之后，整个行业瞬间抛弃了 PPO，全转 DPO 了。
> 效果一样甚至更好，但是难度和成本只有 1/10。

---

### DPO 的核心原理

**数学上的巧妙转化：**
- RLHF 是个复杂的强化学习问题
- DPO 作者数学上证明了：最优解其实可以直接通过简单的分类损失得到
- 不需要奖励模型，不需要 PPO，不需要强化学习
- 就是个普通的监督训练，只是损失函数不一样

**DPO 损失函数的直觉：**

```plaintext
对于同一个 prompt，有两个回答：A（好）、B（坏）

我们要让模型给 A 的概率 尽可能比 B 大

就这么简单。没有奖励模型，没有价值函数，没有 PPO。
就是个二分类问题：A 应该比 B 更可能出现。
```

---

### DPO vs RLHF 全方位对比

| 维度 | RLHF (PPO) | DPO (直接偏好优化) |
| --- | --- | --- |
| **训练阶段** | 3 个阶段（SFT → RM → PPO） | 2 个阶段（SFT → DPO） |
| **工程复杂度** | 极高，十几个人的团队搞半年 | 低，一个人一周就能跑通 |
| **训练稳定性** | 极不稳定，很容易崩 | 非常稳定，和 SFT 一样稳 |
| **效果** | 好 | 一样好，甚至更好 |
| **奖励黑客** | 严重，很容易学坏 | 轻微，好很多 |
| **算力需求** | 极高（PPO 跑几周） | 中等（几小时到几天） |
| **数据需求** | 几十万偏好对 | 几万到十几万偏好对 |
| **可复现性** | 极差 | 极好 |

---

### DPO 的最佳实践

1. **SFT 模型一定要好**
   - DPO 是在 SFT 基础上微调的
   - SFT 不行，DPO 再怎么调也不行
   - SFT 要先达到 80 分，DPO 再提到 90 分

2. **偏好数据质量是王道**
   - DPO 的上限完全由你的偏好数据决定
   - 标注要一致、高质量、覆盖你的目标场景
   - 1 万条高质量偏好 > 10 万条随便标的

3. **超参数很宽容**
   - 学习率：比 SFT 小一个数量级，比如 SFT 用 2e-5，DPO 用 5e-7
   - beta：一般 0.1，控制和原模型的距离，越小越保守
   - 不用怎么调，默认参数一般就很好

4. **别忘了参考模型**
   - DPO 需要一个冻结的参考模型（就是 SFT 模型）
   - 用来算 KL 散度，防止模型跑偏太远

### 对齐项目完整落地路线图

```plaintext
第 1 步：收集 SFT 数据
   ├─ 开源数据集（10 万条）打底
   └─ 收集自己场景的真实数据（1 万条高质量）

第 2 步：SFT 微调
   ├─ 学习率 2e-5，3-5 个 epoch
   ├─ LoRA 先试，效果不好再全参数
   └─ 先做到 80 分

第 3 步：收集偏好数据
   ├─ 让 SFT 模型生成多个回答
   └─ 人工标注哪个更好（几万对）

第 4 步：DPO 对齐
   ├─ 学习率 5e-7，1-2 个 epoch
   ├─ beta=0.1，默认参数就行
   └─ 直接到 90 分

第 5 步：评估和迭代
   ├─ 自动评估 + 人工评估
   └─ 找到不好的 case，加数据继续调
```

---

### 对齐的常见副作用和应对

| 副作用 | 表现 | 怎么应对 |
| --- | --- | --- |
| **对齐税** | 对齐之后，推理、代码能力下降了 | DPO 学习率调小，beta 调小，不要训太久 |
| **过度礼貌** | 不管问什么都先道歉，啰嗦，说废话 | 偏好数据里加简洁回答的偏好 |
| **过度拒绝** | 正常问题也拒绝回答，太保守 | 偏好数据里加"该回答的好好回答"的样本 |
| **模式崩溃** | 所有回答都一个样，没有多样性 | DPO 不要训太多 epoch，KL 散度控制住 |
| **遗忘** | 对齐之后忘了之前会的东西 | 对齐数据里混合一些通用能力的数据 |

## Part 5：Agentic RL 与 GRPO——Agent 专项训练

> 本节内容整理自 Datawhale 开源教程 [hello-agents 第十一章](https://github.com/datawhalechina/hello-agents)，有删改。

### 5.1 为什么 Agent 需要专项训练

前面学的 DPO、SFT 都是通用对齐方法，但** Agent（带工具调用的推理 Agent）** 有独特的训练需求：

| 需求 | 通用 SFT/DPO | Agentic RL |
|------|-------------|------------|
| 多步骤推理 | 无法显式优化中间步骤 | 能优化每步推理质量 |
| 工具调用准确率 | 容易产生幻觉工具调用 | 通过奖励信号专项提升 |
| 探索能力 | 只能模仿训练数据 | 能发现比标注更好的解法 |
| 长程决策 | 难以处理多跳任务 | GRPO 直接优化任务完成率 |

**典型应用场景**：数学推理 Agent、代码生成 Agent、搜索推理 Agent。

### 5.2 GRPO 核心原理

**GRPO（Group Relative Policy Optimization）** = PPO 的简化版，专为 LLM 设计。

**PPO 的问题**：
- 需要单独训练一个 Value Model（价值模型）来估算"未来总收益"
- 训练不稳定，内存占用大
- 对 LLM 来说太重了

**GRPO 的核心思路**：同一个问题，生成多个回答，用"组内相对排名"代替 Value Model：

```
同一个问题 → 模型生成 8 个回答
    ↓
哪个回答对了？（奖励 = 1 或 0）
    ↓
对的回答 → 概率提高
错的回答 → 概率降低
    ↓
不需要 Value Model，用组内均值代替
```

**GRPO 损失函数（简化版）**：

```python
# 对于每个问题，生成 G 个回答
# 奖励：正确的回答 r=1，错误的回答 r=0

# 组内相对奖励
r_i_normalized = r_i - mean(r_group) / std(r_group)

# 策略梯度更新
# 正确的回答 → 增加被选中的概率
# 错误的回答 → 降低被选中的概率
```

**GRPO vs PPO 对比**：

| 对比项 | PPO | GRPO |
|--------|-----|------|
| Value Model | 需要单独训练 | 不需要，用组均值代替 |
| 内存占用 | 高（需要存 Value 网络） | 低（只需策略网络） |
| 训练稳定性 | 中等 | 高 |
| 实现复杂度 | 高 | 低 |
| 适合场景 | 通用 RL | LLM 专项 |

### 5.3 Agentic RL 完整训练流水线

```
┌─────────────────────────────────────────────────────┐
│            Agentic RL 训练流水线                     │
├─────────────────────────────────────────────────────┤
│  Stage 1: 数据准备                                  │
│    - 任务数据集（数学/代码/搜索）                     │
│    - 问题 + 标准答案 / 奖励函数                       │
│                                                     │
│  Stage 2: SFT 微调（行为克隆）                        │
│    - 用高质量 <问题, 正确答案> 对微调                  │
│    - 让模型学会基本推理模式                           │
│                                                     │
│  Stage 3: SFT 评估（可选）                           │
│    - 在测试集上跑一遍，确认基础能力                   │
│                                                     │
│  Stage 4: GRPO 强化学习                             │
│    - 批量生成多个候选回答                            │
│    - 用奖励函数评判                                  │
│    - 组内相对排名更新策略                            │
│                                                     │
│  Stage 5: GRPO 评估                                 │
│    - 对比 PRETRAIN / SFT / GRPO 模型效果             │
└─────────────────────────────────────────────────────┘
```

**奖励函数设计示例**（数学推理场景）：

```python
def reward_function(question: str, response: str, ground_truth: str) -> float:
    """奖励函数：数学推理 Agent"""
    # 1. 格式奖励：有没有最终答案
    has_answer = "### 答案" in response or "The answer is" in response

    # 2. 正确性奖励：答案是否正确
    correct = extract_answer(response) == ground_truth

    # 3. 推理过程奖励：有没有中间步骤
    has_steps = response.count("\n") >= 3  # 至少3行推理

    # 综合奖励
    reward = 0.0
    if has_answer:
        reward += 0.2
    if has_steps:
        reward += 0.2
    if correct:
        reward += 0.6

    return reward
```

### 5.4 Agent 专项训练 vs 通用对齐

| 场景 | 推荐方法 | 原因 |
|------|---------|------|
| 通用对话助手对齐 | DPO / ORPO | 数据易得，不需要专项奖励设计 |
| 数学推理 Agent | GRPO + 答案正确性奖励 | 能优化多步推理过程 |
| 代码生成 Agent | GRPO + 执行结果奖励 | 能学习代码评测反馈 |
| 搜索推理 Agent | GRPO + 任务完成率奖励 | 能优化搜索策略 |
| 工具调用 Agent | GRPO + 调用准确率奖励 | 能减少幻觉调用 |

> **核心认知**：如果你的 Agent 需要在特定领域达到高性能，通用对齐（DPO）只是起点。真正决定上限的，是为你的场景设计的奖励函数和专项训练。
