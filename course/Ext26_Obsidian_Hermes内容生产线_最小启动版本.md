# Ext26：Obsidian + Hermes Agent 内容生产线——最小启动版本

> Obsidian 的强项是存储、链接、沉淀。但它天然不会主动推流程——不会帮你判断资料值不值得收，不会把资料拆成选题，也不会把选题推成草稿。这篇讲的是 Obsidian 做底座、Hermes Agent 做执行层的实际用法。

---

## 核心认知：知识库最大的浪费是"存了不用"

很多人一听"AI + 知识库"就往向量数据库、RAG、知识图谱上想。用下来反而越来越克制。核心盯一件事：

> 这条链路能不能稳定跑通——资料进来 → 判断价值 → 拆成选题 → 生成草稿 → 发布归档

对内容创作者来说，知识库最重要的不是"以后搜得到"，而是"**现在能不能用起来**"。

- 一篇收藏没进过选题池，等于死了
- 一个灵感没变成过大纲，只是心理安慰

这套东西不追求高级，只解决一个具体问题：**让资料别停在资料阶段。**

---

## 目录设计：不是给人看的，是给 Agent 认路的

Vault 结构专门服务内容生产，不是泛用第二大脑：

```
📁 01-Sources      ← 外部资料入库
📁 02-Accounts     ← 各账号的内容资产
   └─ 超级猛
      ├─ Topics    ← 选题池
      ├─ Drafts    ← 草稿区
      └─ Published ← 发布归档
📁 03-Frameworks   ← 内容框架、写作规范、账号定位
```

多了一重考虑：Hermes 能不能读懂？能不能根据目录判断一个文件该进 Source、Topic、Draft 还是 Published？

**最小流转链路先搭出来：**

```
Source Note → Topic Note → Draft Note → Published Note
```

只要这条线跑顺了，知识库就已经从"仓库"变成了"**生产线**"。

---

## 实操第一步：丢资料给 Hermes，做入库判断

碰到一篇文章，不会马上手动搬到 Obsidian。先把链接或正文扔给 Hermes 走第一轮。

**提示词：**
```
判断这篇内容是否值得收入 Obsidian 内容中台。
如果值得，生成 Source Note，包含：
- 核心摘要
- 关键观点
- 适合哪个账号
- 可延展成哪些选题建议
- 建议放入的目录和文件名
```

真正有用的是后面四项——它能判断这篇资料适合哪个账号、能拆出可写的选题、能给出下一步应该进哪条链路。

**普通收藏 vs Hermes 参与的区别：**
- 普通收藏 = 把资料堆进仓库
- Hermes 参与 = 先把资料过一遍"编辑台"，替我做初筛

知识库里最耗精力的不是整理，而是低质量资料太多。什么都收，最后什么都不想看。**第一步不是多存，是让 Agent 帮你挡掉一批不值得推进的东西。**

---

## 实操第二步：拆 Topic Note，把"写什么"钉死

入库之后不急着写正文。让 Hermes 先拆选题。

**提示词：**
```
基于这篇 Source Note，为"超级猛"账号拆 3 个选题。
每个选题包含：
- 一句话定义
- 目标读者
- 用户痛点
- 核心承诺
- 标题候选（3 个）
- 简短提纲
- 判断哪个最适合近期发布
```

同一篇资料可以写成工具测评、方法论、踩坑总结。角度不定，后面越写越散。

**"超级猛"选题四个标准：**
1. 有明确的用户痛点
2. 有真实案例可引用
3. 能讲成可复现的工作流
4. 能给出可操作的方法

比追热点稳定。这个账号不是泛泛聊 AI，而是讲 AI 编程、Agent、工作流和技术管理者提效。一个选题能不能用，看它是落到方法和实践上，而不是热度上。

---

## 实操第三步：生成 Draft，但别让它直接定稿

选题确认后，才让 Hermes 出草稿。

**提示词：**
```
基于这个 Topic Note，写一版适合"超级猛"账号的长文草稿。
要求：
- 从我的使用场景和判断出发，不要写成工具说明书
- 结构清楚，但别像模板
- 每节都有具体做法
- 少用空泛形容词，多写真实问题和取舍
- 结尾给出下一步行动建议
```

**踩过的坑**：AI 工具文最容易写成"某工具是什么、有什么功能、有什么优势、适合谁"。看着很完整，读起来像产品介绍页。

真正有用的写法，是从一个具体问题切入：
- 为什么需要它？
- 之前卡在哪？
- 接进来之后哪一步变了？
- 哪些地方确实省事，哪些地方还需要人判断？

**所以不让 Hermes 一次性给"最终稿"。它的产出我只当初稿用，然后自己过两轮：**
- 第一轮：删掉正确但没信息量的话
- 第二轮：补上真实判断和使用细节

AI 帮你铺结构，但文章有没有可信度，最后看你有没有自己的判断。

---

## 实操第四步：把固定流程写成 Skill

这是 Hermes 比普通聊天工具更适合做内容中台的原因。

如果每次都要重新解释目录结构、账号定位、写作风格，坚持不了一周你就烦了。把稳定流程沉淀成 Skill：

- Source Note 的生成规范
- Topic Note 的字段模板
- "超级猛"的账号定位、选题偏好、表达风格
- 公众号 vs 小红书的差异化要求

以后丢一篇资料进去，不用再讲背景。Agent 知道目录在哪、知道"超级猛"写什么、知道一篇内容要从 Source → Topic → Draft 推进。

**内容生产里最消耗人的，往往不是某一次写作，而是每次都要重新启动上下文：**
- 这个账号写什么？
- 这个选题合不合适？
- 文件放哪？
- 草稿按什么结构？
- 发布后怎么归档？

规则一旦沉淀下来，Agent 才会越来越像**工作流成员**，而不是临时聊天窗口。

---

## 实操第五步：发布后归档，不然系统会断

很多人做内容系统只做到草稿。单独把 Published 拿出来了。每篇发布后补一条归档记录：

```yaml
标题: 
发布时间: 2026-04-28
平台: 公众号
对应Topic: [[Topics/xxx]]
对应Draft: [[Drafts/xxx]]
链接: 
封面文案: 
数据复盘: 
```

这一步的价值不在"记录"，在**反馈**。只看草稿不看发布结果，永远没法判断：
- 什么标题有效
- 什么选题值得续写
- 什么内容适合做系列

**内容系统不能只管生产不管反馈，不然你只是在往前写，没在积累判断力。**

---

## 追求的完整链路

```
Source → Topic → Draft → Published → Review → New Topic
```

这才叫"内容中台"——不是一个资料库，是一个**能持续产生判断的系统**。

---

## 这套方法适合谁

中了三条以上，就很值得试：
- 收藏了大量文章但很少转成输出
- Obsidian 目录越来越复杂，打开频率越来越低
- 同时管多个账号，经常不知道素材该分给谁
- 想让 AI 不只是一次性写作工具，而是参与整个流程管理
- 已经在用 Claude Code、Cursor、Codex 或 Hermes，想把 Agent 接进真实工作流

---

## 最小启动版本

不用一步到位。先搭这个：

```
📁 01-Sources
📁 02-Topics
📁 03-Drafts
📁 04-Published
```

然后固定三个提示词：
1. **入库判断** — 请把这篇资料整理成 Source Note，判断是否值得继续推进
2. **拆选题** — 基于这篇 Source Note，为我的账号拆 3 个可写选题，推荐最适合近期发布的一个
3. **出草稿** — 基于这个 Topic Note，写一篇有真实使用场景、有方法步骤、有个人判断的草稿

先把这三步跑顺，就能明显感觉到：**知识库不再是你一个人手动维护的仓库，它开始变成一条有人帮你推着走的生产线。**

---

## 最大的感受

> 别只把 AI 当写作工具，把它放进流程里。让它只写一篇文章，价值是一次性的。让它参与选题、归档、草稿、发布复盘，价值才会累积。
